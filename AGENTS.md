## Project Overview

ELENOR is a **documentation-only architecture repository** for a future AI accelerator targeting inference and light training. There is **no source code, no build system, no test suite, no CI** — the deliverable is a set of ~28 interlocking Markdown design specs plus a PDF export script.

The accelerator's central thesis is **`Compute != Control != Data Movement`**: hardware consumes low-level objects (command buffers, descriptors, TileGroupTasks, Tile Programs), never high-level graphs. Four engines cover the workload space:

| Engine  | Full name                       | Responsibility                 | Target workloads                                                          |
| ------- | ------------------------------- | ------------------------------ | ------------------------------------------------------------------------- |
| **BOA** | Block Outer-product Accelerator | Dense compute                  | GEMM, Conv, QK, AV, Expert MLP                                            |
| **EVU** | Enhanced Vector Unit (EVU-MT)   | Irregular / vector compute     | Softmax, Norm, RoPE, Activation, Gather/Scatter, tail/mask                |
| **MFE** | Memory Flow Engine              | Data movement / stream shaping | Page Stream, Segment Stream, Sparse Block, Persistent Memory              |
| **USE** | Unified State Engine            | Stateful / control compute     | Scan, recurrence, dynamic-shape assist, token routing, checkpoint/restore |

Target configs scale edge → datacenter: **Edge** (8–16 tiles, LPDDR), **Balanced** (64 tiles, HBM/DDR), **High End** (128 tiles, HBM).

## Architecture & Data Flow

### Hardware hierarchy

```text
Host / System SoC
   └─ Host Interface (PCIe/CXL/AXI) ── Runtime Processor (RISC-V/uCtrl) ── Global PMU
        └─ Global Scheduler / Command Queue / Event Fabric
             ├─ Global DMA (2D/strided/copy)
             ├─ Memory Controller (HBM/DDR/LPDDR)
             └─ Collective Engine (reduce/broadcast)
                  └─ NoC / Router ── Tile Group × N
                       └─ Compute Tile × 8 (per group)
                            ├─ Tile UCE (control)
                            ├─ L1 SRAM + Tile DMA
                            ├─ BOA cluster (OPA × 4)
                            ├─ EVU
                            ├─ MFE tile port
                            ├─ USE
                            └─ local event / PMU
```

- **OPA** (Outer-product Primitive Array) is the BOA's smallest dense-compute primitive — _not_ a runtime-schedulable engine. Tile UCE launches a BOA descriptor; the BOA sequencer lowers it into OPA micro-loops.
- **Tile UCE** (Unified Control Engine) and **USE** are two functional components that may share one tile-local RISC-V / micro-controller. UCE owns program control, engine launch, event wait, stream token, descriptor patch; USE owns state register/cache, scan, recurrence, checkpoint/restore.

### Control-flow hierarchy: `Graph → Group Task → Tile → Engine`

```text
Graph Schedule PC / Group Task Iterator   (Device Runtime)
   └─ Tile Group Sequencer action index   (Tile Group Sequencer, per Tile Group)
       └─ Tile PC / Tile UCE             (per Compute Tile)
           └─ BOA / EVU / MFE / USE tasks
               └─ BOA / EVU micro-sequencer + datapath
```

| Control layer | Controller           | Managed objects                                             |
| ------------- | -------------------- | ----------------------------------------------------------- |
| Graph         | Device Runtime       | graph schedule, group task dependency, context, queue       |
| Group         | Tile Group Sequencer | Group Task, Group DMA, role dispatch, group barrier         |
| Kernel        | Tile UCE             | Tile Program, stream token, descriptor patch, engine launch |
| State         | USE                  | state update, scan, recurrence, checkpoint/restore          |
| Compute       | BOA / EVU Sequencer  | micro loop, operand fetch, local reduce, vector mask        |

### Data-flow hierarchy

`HBM/DDR/LPDDR → L2 (Group SRAM via Group DMA / MFE) → L1 (Tile DMA / MFE tile port) → Engine (BOA/EVU/USE)`, with `Role → Role` flow through Stream Queues and `Tile → Tile` flow through Collective/Broadcast.

### Execution objects & program residency

A compiled package (`model.pkg`) is **not** a graph or an op — it is `graph_schedule.bin` + `group_tasks/` + `tile_programs/` + `descriptors/` + `weights/` + `relocation_table`. Core objects:

| Object          | Granularity        | Role                                                     |
| --------------- | ------------------ | -------------------------------------------------------- |
| Graph Schedule  | whole graph        | group task dependency, context, queue, memory lifetime   |
| TileGroupTask   | Tile Group         | group task, role dispatch, HBM→L2 DMA                    |
| TileRoleBinding | role               | role_id, tile_mask, Tile Program, in/out stream          |
| Tile Program    | Tile               | Tile UCE executes: L2→L1 DMA, BOA/EVU/MFE/USE            |
| Descriptor      | dynamic params     | shape, stride, address, tiling, stream, state, patch     |
| Stream Queue    | inter-role         | producer-consumer, credit, backpressure, EOS/error token |
| Slot Frame      | Tile L1            | L1 memory binding for tile program + descriptor          |
| Program Table   | residency registry | program_id → section metadata / local handle             |
| Event Table     | runtime/group/tile | completion, dependency, fault, timeout                   |

**Residency contract:** compiler never emits `program.load/prefetch/install`; runtime only registers metadata; Tile Group Sequencer guarantees program-ready before dispatch; Tile UCE only consumes a resident local handle. **Cold Launch** = residency miss → implicit fetch/verify/install. **Warm Launch** = program resident → patch descriptor/context/shape only. Correctness is gated by `program_id + version + hash + epoch`.

### Tile-SPMD programming model

One Tile Program template runs on many tiles; per-tile data is distinguished by `tile_id`, `group_id`, descriptor offset, and slot-frame binding (analogous to CUDA SPMD). The compiler emits MLIR-like IR across six non-overlapping dialects:

| Dialect          | Semantics                                                    | Output                  |
| ---------------- | ------------------------------------------------------------ | ----------------------- |
| `elenor.boa`     | dense GEMM/Conv/QK/AV                                        | BOA descriptor template |
| `elenor.evu`     | elementwise, softmax, norm, mask/tail, gather/scatter subset | EVU descriptor template |
| `elenor.mfe`     | Page/Segment Stream, layout/reorder                          | MFE stream descriptor   |
| `elenor.use`     | state, scan, recurrence, checkpoint/restore                  | USE state descriptor    |
| `elenor.runtime` | command, event, barrier, branch_shape, launch_group_task     | command template        |
| `elenor.package` | section, relocation, manifest, kernel binding                | executable package      |

Dialects must not encroach on each other (BOA dialect never expresses event-wait; runtime dialect never expresses matmul tile layout).

### Module specs (`design/elenor_*`)

Every module lives at `design/elenor_<module>/ELENOR_<Module>_Design.md`. The 27 module dirs and their scope:

- `elenor_runtime_abi` — runtime Command/Event ABI v0, object model, fault record
- `elenor_host_interface` — PCIe/CXL/AXI, doorbell, queue config, interrupt protocol
- `elenor_global_scheduler` — command pipeline, event scoreboard, barrier manager, timeout wheel
- `elenor_pmu` — layered PMU, counter lifecycle, unique stall attribution
- `elenor_fault_reset` — fault handling pipeline, reset domains, drain/cancel

**Compute & engine layer**

- `elenor_boa` — dense compute engine (BOA)
- `elenor_evu` — irregular/vector engine (EVU-MT; note an unused `ELENOR_EVU_Design copy.md` exists — **use the non-copy version**)
- `elenor_mfe` — memory flow engine (4 modes: Page/Segment/Sparse/Persistent Stream)
- `elenor_use` — unified state engine (scan/recurrence/checkpoint)
- `elenor_collective` — reduce/broadcast/all-reduce/reduce-scatter
- `elenor_opa` — BOA's outer-product primitive (internal to BOA, not runtime-schedulable)

**On-chip organization & data flow**

- `elenor_tile_group` — Tile Group: group task FSM, shared SRAM, group DMA
- `elenor_compute_tile` — Compute Tile: kernel execution domain, L1 SRAM, engine arbitration
- `elenor_tile_uce` — Tile UCE: Tile Program PC, engine launch, descriptor patch
- `elenor_slot_frame` — L1 SRAM layout, frame bind, descriptor patch FSM
- `elenor_memory_noc` — NoC router pipeline, DMA descriptor, SRAM bank arbiter
- `elenor_global_dma` — 1D/2D/strided copy, descriptor lifecycle, burst splitter
- `elenor_stream_queue` — producer-consumer queue, credit/backpressure, EOS propagation
- `elenor_tile_group_sequencer` — Tile Group Sequencer action index, role dispatch, wait FSM

**Software stack & system**

- `elenor_compiler` — MLIR dialect strategy, pass pipeline, descriptor templates
- `elenor_executable_package` — package binary layout, cold/warm load/patch/submit
- `elenor_driver_firmware` — user runtime / kernel driver / firmware split, doorbell protocol
- `elenor_workload_mapping` — per-workload mapping (Dense/Paged/MoE/SSM/GNN/multi-model)
- `elenor_verification_bringup` — verification tiers, bring-up FSM, phase exit criteria
- `elenor_physical_timing_power` — physical layer, clock/reset/power, timing closure

## Development Commands

There is no build, test, or lint — only **PDF export**.

### Generate a PDF from a design doc

```bash
# Requires Docker (image: alwaysproblem/pandoc) + xelatex engine + eisvogel template
bash scripts/generate_pdf.sh -f design/elenor_compute_tile/ELENOR_Compute_Tile_Design
# Flags:
#   -f, --file <path>      markdown file (extension optional; .md appended)
#   -e, --pdf-engine <eng>  default xelatex
#   -N, --numbers          add chapter numbers to headers
#   -nl, --nolistings      disable --listings
```

The script runs `docker run --rm -v "$(pwd):/data" alwaysproblem/pandoc …` with `--template gen_docs_config/eisvogel.latex --pdf-engine=xelatex --shift-heading-level-by=-1`. Work from the repo root so the relative template path resolves.

### Working with the docs

- No package manager, no language runtime required to read/edit the design docs — they are plain Markdown.
- Diagrams: `design/image/` holds `.png` (rendered figures), `.svg`, and `.drawio` (sources). Docs embed PNGs via `![alt](./image/<name>.png)`.

## Code Conventions & Common Patterns

The design docs follow a **strict, shared 9-section template**. Every module spec (`elenor_*/*_Design.md`) uses these exact headings:

1. `## 1. 定位、目标和 First Silicon cutline` — positioning, goals, V1 cutline table (must-close vs. deferred)
2. `## 2. 职责、非职责和 ownership` — responsibilities, non-responsibilities, ownership matrix
3. `## 3. 微架构和状态机` — microarchitecture, FSMs (often sub-`###` per state machine)
4. `## 4. 接口、descriptor、寄存器和协议` — interfaces, descriptor/CSR/ABI drafts (`v0` suffix), protocol
5. `## 5. 数据流、控制流和时序路径` — data/control/timing flows
6. `## 6. 配置、PPA、性能模型和 PMU` — config params, PPA, roofline, PMU counters
7. `## 7. RTL/软件实现建议` — RTL/software implementation guidance
8. `## 8. 验证、bring-up 和验收标准` — verification, bring-up order, acceptance criteria
9. `## 9. 风险、取舍和后续细化方向` — risks, tradeoffs, follow-ups

### Design doc conventions

- **Language:** Primary content is Chinese; technical terms (BOA, EVU, descriptor, TileGroupTask, FSM, etc.) stay in English. Keep this mixed style.
- **Unfrozen values:** Any not-yet-decided number/encoding is explicitly marked with one of `由后续规格冻结` / `由 SRAM profile 冻结` / `由 PPA exploration 冻结` ("to be frozen by later spec / SRAM profile / PPA exploration"). **Never fabricate binary encodings or physical macros** in a spec — defer them with this marker.
- **ABI versioning:** All interface drafts are tagged `v0` (e.g. `DMA descriptor v0`, `ABI v0 command layout`, `fault record v0`, `binary ABI v0`). Treat `v0` as draft, not frozen.
- **Descriptor templates:** Use C-style `typedef struct { … } elenor_*_template_v0_example_t;` with a `patch_mask` field declaring which fields runtime/Tile UCE/MFE/USE may patch. Undeclared fields must not be modified at runtime.
- **Ownership tables:** Module boundaries are explicit — each spec lists what it owns, what it does **not** own, and an ownership matrix. When touching a module, respect and update these boundaries; do not silently expand a module's scope.
- **Diagrams:** ASCII art in fenced `text` blocks for structural/pipeline diagrams; PNG images for rendered architecture figures; mermaid is not used.
- **Cross-references:** Specs reference siblings by module name (e.g. "见 `elenor_compute_tile`"). The master doc (`ELENOR_Architecture_Design_v1.md`) is the canonical integration point and indexes all modules.
- **Workload mapping tables:** Each engine doc's `## 8` / `## 5` sections map concrete workloads (Dense Attention, Paged Attention, MoE, SSM/Mamba/RWKV, multi-model) to engine roles.

### Python conventions (`pipeline_validator/`)

The validator follows PEP 8 enforced by **ruff** (configured in `ruff.toml`)
and static type-checking with **mypy**. Both run as pre-commit hooks —
commits are blocked on lint or type errors.

| Aspect        | Setting                       | Source                                     |
| ------------- | ----------------------------- | ------------------------------------------ |
| Line length   | **108** characters            | `ruff.toml` `line-length`                  |
| Indent        | **2 spaces** (not 4)          | `ruff.toml` `indent-width`                 |
| Quotes        | **double** (`"`)              | `ruff.toml` `[format]`                     |
| Line endings  | **LF**                        | `ruff.toml`                                |
| Python target | **3.12**                      | `ruff.toml` `target-version`               |
| Ruff rules    | E, F, W, I, N, UP, B, C4, RUF | `ruff.toml` `[lint]` `select`              |
| Ruff ignores  | N801 etc.                     | avoids renaming arch names like `DMA_DESC` |
| Type checking | **mypy** (zero errors)        | `.pre-commit-config.yaml`                  |
| Auto-format   | ruff `--fix` + prettier       | `.pre-commit-config.yaml`                  |

**Do**:

- Use `from __future__ import annotations` at the top of every `.py` file.
- Add `assert self.program is not None` before accessing `self.program.*`
  when the type system sees it as `Optional`.
- Prefix unused unpacked variables with `_`: `_desc_id, _dst, event_id = ins.args`
- Keep imports sorted: ruff's `I` rules auto-sort them.
- Use `# noqa: F401` on public-API imports only referenced in `__all__`.
- Run `conda run -n elenor-validator pre-commit run -a` before committing.

**Avoid**:

- Don't suppress ruff rules globally without a comment explaining why.
- Don't suppress mypy with `# type: ignore` unless architecturally necessary.
- Don't use 4-space indent — follow the project's 2-space convention.

## Runtime/Tooling Preferences

### For the validator (`pipeline_validator/`)

- **Runtime:** Python 3.11 via conda environment `elenor-validator`.
  ```bash
  conda env create -f pipeline_validator/environment.yml
  conda activate elenor-validator
  ```
- **Formatter:** ruff (`ruff.toml` at repo root). `ruff check --fix` auto-fixes
  imports, formatting, and most lint issues. Prettier handles Markdown/JSON/YAML.
- **Type checker:** mypy (zero errors required; run via pre-commit).
- **Pre-commit:** `conda run -n elenor-validator pre-commit run -a` runs the
  full gate before commit. Installed hooks: ruff, mypy, prettier,
  trailing-whitespace, end-of-file-fixer, check-yaml.
- **Package manager:** pip within the conda environment; no `pyproject.toml` yet.
- **PDF export:** Docker image `alwaysproblem/pandoc`, Pandoc with `xelatex`,
  `eisvogel.latex` template at `gen_docs_config/eisvogel.latex`.
  Run `bash scripts/generate_pdf.sh` from the repo root.

### For the design docs

- **No runtime / package manager** needed. Only Docker for PDF export.
- **No `.gitignore`** at repo root; no CI config; no Makefile.

## Testing & QA

### Validator tests (`pipeline_validator/`)

Run with pytest in the conda environment:

```bash
conda run -n elenor-validator python -m pytest pipeline_validator/tests/ -v
```

- **26 tests** across 4 test classes:
  - `TestStreamQueue` (9 tests) — credit invariant, backpressure, EOS, reset.
  - `TestIR` (5 tests) — Tile/TileGroupTask builders produce valid IR.
  - `TestSimulation` (9 tests) — end-to-end: all 6 workloads complete,
    credit invariant holds, report is renderable.
  - `TestTracer` (3 tests) — Perfetto/Chrome trace JSON + HTML output.
- **All tests must pass** before committing. Run `pre-commit run -a` first,
  then `pytest`.

### Design doc review

The document review in `review/ELENOR_Document_Review.md` identified:

- **8 P0 blockers** — shared ABI/contract not frozen, all docs still use "`样例`" (example) structures, Tile Program residency path unfrozen, Paged-Attention cross-engine ABI unfrozen, event-wait lacks sequence/generation fields, Fault Record has two incompatible `v0` definitions, BOA split-K semantics unclosed, EVU V1 cutline contradicts acceptance plan, USE checkpoint/event-assist deferred but V1 depends on it.
- **10 cross-document consistency issues** — command-type enum conflicts, event-status encoding conflicts, fault-record struct conflicts, descriptor naming inconsistency, PMU counter naming fragmentation, stall-attribution layer mismatch, address-reference model inconsistency, context-ID width inconsistency, event-ID width inconsistency, Stream Queue semantics unfrozen.

**Guidance for an AI assistant:** when asked to extend or refine the architecture, prioritize **freezing/aligning contracts** (the P0 + consistency items) over adding new features. When editing a spec, keep the shared 9-section template, the `v0` ABI tagging, the `由后续规格冻结` deferral markers, and the ownership/non-ownership boundaries intact. Cross-check any ABI/descriptor change against `review/ELENOR_Document_Review.md` §3–§4 to avoid widening the consistency gaps.
